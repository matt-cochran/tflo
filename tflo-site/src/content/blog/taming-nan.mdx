NaN is silent. It doesn't crash your program. It just makes your outputs meaningless.

IEEE 754 floating point propagates NaN through every operation. One bad input — a division by zero, a missing sensor reading, a log of a negative number — and every downstream computation becomes NaN too. Your crossover detector fires Nothing. Your SMA vanishes. Your alerts go quiet.

The worst part: nothing tells you. The graph keeps running, producing NaN after NaN, and you don't notice until someone asks why no signals fired today.

tflo's validation system gives you control over this, with the understanding that some features are fully implemented while others are planned.

## Permissive by default

Out of the box, `ValidationOptions::new()` is permissive. It allows NaN, allows infinity, doesn't check timestamps. This matches IEEE 754 semantics — NaN propagates naturally, and your code runs without overhead.

```rust
use tflo_core::validation::ValidationOptions;

let options = ValidationOptions::new();
// assert_sorted: false
// reject_nan: false
// reject_inf: false
// error_on_nan: false
// error_on_inf: false
// error_on_negative: false
// min_warmup: 0
// max_gap_ms: None
```

This is fine for development. You're testing against clean data, you know your inputs are valid, and you don't want the overhead of checks. But for production, you probably want more.

## Strict mode

One call flips all the switches:

```rust
let options = ValidationOptions::strict();
// assert_sorted: true
// reject_nan: true
// reject_inf: true
// error_on_nan: true
// error_on_inf: true
// error_on_negative: true
// min_warmup: 1
// max_gap_ms: None
```

When you use strict mode with the `validated()` method on iterators, the current implementation checks `assert_sorted` — ensuring timestamps are monotonically non-decreasing:

```rust
use tflo_core::prelude::*;
use tflo_core::validation::ValidationOptions;

let results: Vec<TFloResult<f64>> = events.into_iter()
    .validated(ValidationOptions::strict(), |t| {
        t.timestamp(|x| x.ts);
        let value = t.prop(|x| x.value);
        value.sma(20_u64.secs())
    })
    .collect();
```

Note the return type: `TFloResult<f64>` (an alias for `Result<f64, TFloError>`) instead of `f64`. Each result is either `Ok(f64)` or `Err(TFloError)`. If `assert_sorted` is enabled and timestamps arrive out of order, the iterator returns `Err(TFloError::OutOfOrderTimestamp)`.

**Important:** The option fields `reject_nan`, `error_on_nan`, `reject_inf`, `error_on_inf`, `error_on_negative`, and `min_warmup` are stored in `ValidationOptions` but currently only `assert_sorted` is checked during the validated iterator step. NaN/Inf handling via the validated iterator is planned for a future release. For now, you should sanitize your data upfront.

## Sanitizing input with standalone validators

Use the standalone validation functions from `tflo_core::validation` as a separate step before feeding data into `.tflo()`:

```rust
use tflo_core::validation::{require_finite, require_not_nan};
use tflo_core::prelude::*;

let results: Vec<TFloResult<f64>> = events.into_iter()
    .filter_map(|event| {
        // Validate inputs before they reach the graph
        require_not_nan(event.value).ok()?;
        require_finite(event.value).ok()?;
        Some(event)
    })
    .tflo(|t| {
        t.timestamp(|x| x.ts);
        let value = t.prop(|x| x.value);
        value.sma(20_u64.secs())
    })
    .collect();
```

This gives you explicit control: bad values are filtered out before the pipeline sees them, and the graph only processes clean data.

## Builder pattern for fine control

You don't have to go all-or-nothing. `ValidationOptions` uses a builder pattern so you can pick exactly which checks to enable:

```rust
let options = ValidationOptions::new()
    .assert_sorted(true)          // Catch out-of-order timestamps
    .reject_nan(true)             // (stored, planned for future)
    .min_warmup(10);              // (stored, planned for future)

let options = ValidationOptions::new()
    .error_on_nan(true)           // (stored, planned for future)
    .error_on_inf(true)           // (stored, planned for future)
    .max_gap_ms(30_000);          // (stored, planned for future)
```

The individual methods — `assert_sorted()`, `reject_nan()`, `error_on_nan()`, `reject_inf()`, `error_on_inf()`, `error_on_negative()`, `min_warmup()`, `max_gap_ms()` — are all `const fn`, so you can build the options at compile time with zero runtime cost.

## What each flag stores

- **`reject_nan`** — intended to silently drop records that produce NaN. Currently stored but not checked during iteration.
- **`error_on_nan`** — intended to return an error on NaN steps. Currently stored but not checked during iteration.
- **`reject_inf` / `error_on_inf`** — same logic for infinity. Stored but not yet enforced.
- **`error_on_negative`** — intended to catch negative values in operations that don't support them. Stored but not yet enforced.
- **`assert_sorted`** — fully implemented. Checks that timestamps are monotonically non-decreasing. Violations return `Err(TFloError::OutOfOrderTimestamp)`.
- **`min_warmup`** — intended to require a minimum number of records before any output is considered valid. Stored but not yet enforced.
- **`max_gap_ms`** — intended to fail on gaps exceeding this threshold. Stored but not yet enforced.

## The error types

On the validated iterator, errors are wrapped in `TFloError`. The currently active variant is:

```rust
pub enum TFloError {
    OutOfOrderTimestamp,
    // Additional error variants are defined but only
    // OutOfOrderTimestamp is actively raised by validated()
}
```

Additional `ComputeError` variants (such as `NaN`, `Infinite`, `DivisionByZero`, `InvalidInput`) are defined in the codebase and can be used in custom nodes, but are not automatically raised by the validated iterator today.

## When to use each mode

- **Development and testing**: `ValidationOptions::new()` — permissive, fast, no noise.
- **CI and pre-production**: `ValidationOptions::strict()` with upfront sanitization — filter NaN/Inf with `require_not_nan`/`require_finite` before the graph, and let `assert_sorted` catch ordering issues.
- **Production with monitoring**: `ValidationOptions::new().assert_sorted(true)` plus standalone validators — catch ordering issues automatically, sanitize values explicitly.
- **Legacy data sources**: filter with `require_not_nan` and `require_finite` before the pipeline, and optionally enable `assert_sorted`.

The point isn't to always use strict mode. The point is to choose. Silence is the enemy of debuggable pipelines — decide what you want to catch, and catch it explicitly.
